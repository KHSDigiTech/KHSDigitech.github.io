Testing Table Template and Guide

Systematic testing for your AS92004 Contact Book assessment

Purpose

A testing table is a crucial tool for proving that your program works as expected. It helps you think systematically about what to test and provides clear evidence of your program's functionality, robustness, and handling of edge cases. You should fill this out as you develop and test your program.

How to Use This Table

  • Test Case ID: A unique number for each test (e.g., 1, 2, 3...).
  • Feature Being Tested: What part of your program are you testing? (e.g., "Add Contact", "Main Menu Choice").
  • Test Type: What kind of input are you testing?
    • Expected: The program is working under normal, expected conditions.
    • Boundary: The program is working at the limits of its conditions (e.g., an empty list, the highest valid menu option).
    • Invalid: The program is being tested with input it should reject (e.g., text instead of a number, a menu option that doesn't exist).
  • Input Data: Exactly what you will type into the program for this test.
  • Expected Outcome: What should happen when you provide the input?
  • Actual Outcome: What actually happened when you ran the test?
  • Pass/Fail: Did the actual outcome match the expected outcome?
  • Action Taken (if Fail): If the test failed, what did you do to fix it? (e.g., "Fixed the if statement to correctly check for the number 3.", "Added a try-except block to handle text input.")

Testing Strategies

1. Test Types You Should Include

  • Expected Cases: Normal usage - valid menu choices, typical contact data
  • Boundary Cases: Edge conditions - empty lists, maximum/minimum values, first/last menu options
  • Invalid Cases: Error conditions - wrong data types, out-of-range numbers, empty inputs

2. Features to Test

  • Main menu navigation (all valid options)
  • Add contact functionality
  • View contacts functionality
  • Program exit
  • Error handling for invalid inputs
  • Edge cases (empty contact book, etc.)

3. Assessment Evidence

For NCEA Assessment: Include screenshots or written logs of your testing process. Show both passing and failing tests, and document how you fixed any failures. This demonstrates your systematic approach to ensuring program quality.

Tips for Effective Testing

  • Test as you build: Don't wait until the end to test everything
  • Be specific: Record exact input and output, don't just say "it worked"
  • Test failure scenarios: Your program should handle errors gracefully
  • Document fixes: Show how you improved the code based on test results
  • Retest after fixes: Make sure your changes actually work
  • Keep evidence: Screenshots or log files prove your testing process
